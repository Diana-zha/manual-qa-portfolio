# Contract Testing Checklist

## Endpoint: GET /pet/{petId}

### Request

- [ ] Проверить, что используется HTTP method `GET`.
- [ ] Проверить, что запрос отправляется на endpoint `/pet/{petId}`.
- [ ] Проверить, что параметр `petId` передаётся в path.
- [ ] Проверить, что параметр `petId` является обязательным.
- [ ] Проверить запрос с корректным существующим `petId`.
- [ ] Проверить запрос с несуществующим `petId`.
- [ ] Проверить запрос с нечисловым значением `petId`.
- [ ] Проверить запрос с отрицательным `petId`.
- [ ] Проверить запрос с `petId = 0`.
- [ ] Проверить запрос со значением `petId`, превышающим диапазон `int64`.

### Successful response — 200 OK

- [ ] Проверить, что для существующего питомца возвращается `200 OK`.
- [ ] Проверить, что response header `Content-Type` соответствует `application/json`.
- [ ] Проверить, что response body является JSON object.
- [ ] Проверить соответствие response body схеме `Pet`.
- [ ] Проверить наличие обязательного поля `name`.
- [ ] Проверить, что поле `name` имеет тип `string`.
- [ ] Проверить наличие обязательного поля `photoUrls`.
- [ ] Проверить, что поле `photoUrls` имеет тип `array`.
- [ ] Проверить, что каждый элемент `photoUrls` имеет тип `string`.
- [ ] Если поле `id` присутствует, проверить, что оно имеет тип `integer`.
- [ ] Если поле `category` присутствует, проверить соответствие схеме `Category`.
- [ ] Если поле `tags` присутствует, проверить, что оно имеет тип `array`.
- [ ] Если поле `tags` присутствует, проверить соответствие каждого элемента схеме `Tag`.
- [ ] Если поле `status` присутствует, проверить, что оно имеет тип `string`.
- [ ] Если поле `status` присутствует, проверить, что значение входит в enum: `available`, `pending`, `sold`.

### Error responses

- [ ] Проверить возврат `400 Bad Request` для некорректного `petId`.
- [ ] Проверить возврат `404 Not Found` для несуществующего питомца.
- [ ] Проверить фактический response body для `400`, так как схема ошибки в контракте не указана.
- [ ] Проверить фактический response body для `404`, так как схема ошибки в контракте не указана.
- [ ] Проверить поведение API при неожиданной серверной ошибке.
- [ ] Проверить, соответствует ли фактический неожиданный ответ секции `default`.

### Security

- [ ] Проверить запрос без `api_key`.
- [ ] Проверить запрос с валидным `api_key`, если тестовые данные доступны.
- [ ] Проверить, соответствует ли фактическое требование авторизации секции `security`.
- [ ] Зафиксировать расхождение, если endpoint работает без авторизации, хотя она указана в контракте.

### Contract quality

- [ ] Проверить, что обязательные поля `name` и `photoUrls` действительно всегда возвращаются.
- [ ] Проверить, не возвращаются ли поля с типами, отличающимися от контракта.
- [ ] Проверить наличие дополнительных полей, не описанных в схеме.
- [ ] Проверить, совпадают ли фактические status codes с описанными в OpenAPI.
- [ ] Зафиксировать неоднозначность: для `400` и `404` не описана схема response body.
